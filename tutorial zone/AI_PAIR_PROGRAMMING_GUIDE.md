# AI 对话工作流教程：零代码构建可扩展节点编辑器

本教程展示了如何通过 **纯对话** 与 AI（如 Claude、GPT-4 或其它代码助手）协作，**无需亲自编写一行代码**，从零构建出一个架构清晰、可扩展的 React 节点编辑器。

我们以实际项目 `reactflow-mod-studio` 的诞生过程为蓝本，提取出可复用的对话模式和工作流阶段。你只需将本教程作为剧本，与 AI 一步步对话即可。

---

## 🧠 核心理念：AI 是架构师 + 工匠，你是产品经理 + 测试员

- **你**负责描述需求、确认设计、验证结果。  
- **AI** 负责架构设计、输出完整代码、解释原理。  
- 每次只前进一步，确认无误后再继续，确保项目始终可运行。

---

## 📂 准备工作：项目快照

在每次对话开始前，请先运行项目快照脚本，将完整项目内容和目录结构导出为一个文件。  
脚本已提供（`collect_react_info.py`），运行后生成 `react_project_snapshot.txt`。

**在对话中，将此文件内容完整粘贴给 AI**，让 AI 完全了解当前项目状态。这是精准协作的基础。

---

## 🚀 工作流阶段总览（本项目已全部完成）

| 阶段 | 目标 | 主要对话模式 |
|------|------|--------------|
| 1️⃣ 现状评估 | AI 分析现有代码质量、架构，给出改进建议 | 粘贴快照 → AI 输出分析报告 |
| 2️⃣ 架构重构 | 将单体逻辑拆分为事件总线 + Mod 插件 | 提出目标 → AI 设计方案 → 分步实现文件 |
| 3️⃣ Bug 修复 | 解决重构引入的卡死、重复事件等问题 | 粘贴错误日志 → AI 分析根因 → 输出修复代码 |
| 4️⃣ 类型安全 | 消除 `any`，为接口和组件添加精确类型 | 逐文件修改 → 每次 `typecheck` 验证 |
| 5️⃣ 功能增强 | 完善剪贴板、全选、连接交互等细节 | 描述期望行为 → AI 给出修改代码 |
| 6️⃣ 扩展机制 | 设计自定义 Mod 插槽、节点模板注册中心 | 描述扩展需求 → AI 设计 API → 实现文件 |
| 7️⃣ 全面注释 | 所有文件添加详细中文注释 | 分模块请求 → AI 逐个输出注释版代码 |
| 8️⃣ 文档编写 | README、API 参考、Mod 教程、本工作流指南 | 提出文档需求 → AI 输出完整 Markdown |

> 注意：本项目已完成上述所有阶段，以下各节提供了每个阶段的具体对话范例，供你参考或复用于其他项目。

## 1️⃣ 现状评估（示例对话）

**你**：  
> 这是项目的完整快照报告（粘贴 `react_project_snapshot.txt`）。请分析项目结构、代码质量、潜在问题，并给出重构建议。

**AI**：  
> （输出分析报告，指出耦合点、重复代码、类型缺失、事件驱动缺失等，并建议采用事件总线 + Mod 架构。）

**验证**：阅读 AI 报告，确认理解当前项目状态。

---

## 2️⃣ 架构重构（事件总线 + Mod）

这是最关键的一步，需要分多次对话完成。以下为本项目实际使用的对话序列。

### 2.1 设计方案
**你**：  
> 我要将项目改造成事件总线 + Mod 插件架构，类似游戏引擎的 ECS。请给出详细方案，包括 EditorEvent 类型、EditorBus 接口、Mod 注册机制，以及如何将现有功能拆分为独立 Mod。

**AI**：  
> （输出 `EditorEvent` 联合类型、`EditorBus` 接口、`useEditorBus` 实现、`EditorMod` 接口、`initMods` 函数设计，并列出应拆分的 Mod 列表：历史、批量连线、对齐、剪贴板、重连、项目配置等。）

### 2.2 分步实现
**模式**：每步 AI 给出 1-2 个**完整覆盖文件**的代码，你粘贴后运行 `npm run typecheck`，确认无误再继续。

**对话示例**：
> **你**：第一步，创建 `src/bus/types.ts` 和 `src/bus/useEditorBus.ts`，给出完整代码。  
> **AI**：（输出完整代码）  
> **你**：已保存，typecheck 通过。下一步，创建 `mod-history.ts`。  
> **AI**：（输出完整代码）  
> ...（重复直到所有内置 Mod 完成）

**关键技巧**：
- 始终要求 AI 输出**完整文件内容**，而非片段。
- 每完成一个文件，立即运行 `typecheck`。
- 遇到报错，将错误信息全文粘贴给 AI，它通常能迅速修复。

## 3️⃣ Bug 修复（示例）

重构后出现卡死、事件重复、交互异常等 bug。

**对话模式**：
1. 启动项目，打开浏览器控制台。
2. 复制完整的错误日志和调用栈。
3. 粘贴给 AI，附上简单描述（如“放下节点后界面卡死”）。
4. AI 分析原因并提出修复方案，输出修改后的文件。

**示例**：
> **你**：（粘贴重复事件日志）每次添加节点都触发两次 NODE_ADDED，界面越来越卡。  
> **AI**：原因是 ID 计数器未同步，导致新节点 ID 冲突。修复方法：在 reducer 中添加去重，并在加载工作流时同步计数器。以下是修改后的 `useEditorBus.ts` 和 `utils/index.ts`...

**常见修复主题**：
- 历史记录栈管理错误（需要点两次撤销）
- 重连时菜单意外弹出（抑制计数器 + `setTimeout` 归零）
- 节点模板注册中心与覆盖机制冲突

## 6️⃣ 扩展机制设计

让项目成为真正的“框架”。

### 6.1 自定义 Mod 插槽
**对话**：
> **你**：我想让其他人能编写 Mod 而不用改核心代码。请设计一个 `custom-mods/` 目录和注册机制。  
> **AI**：（输出 `custom-mods/index.ts`, `mods/index.ts` 修改, `App.tsx` 导入, 教程文档）

### 6.2 节点模板注册中心
**对话**：
> **你**：节点模板也应该可动态注册。请设计一个注册中心，并提供添加/覆盖模板的示例 Mod。  
> **AI**：（输出 `registry/nodeTemplateRegistry.ts`, 修改所有引用文件, 示例 Mod）

### 6.3 覆盖与降级
**对话**：
> **你**：内置 Mod 应该可以被用户自定义 Mod 覆盖，且如果自定义 Mod 初始化失败，应该自动回退到内置版本。请实现这个机制。  
> **AI**：（输出修改后的 `mods/index.ts`，添加覆盖映射和 try-catch 降级）

## ✅ 验证循环

每次 AI 输出代码后，立即执行：

```bash
npm run typecheck   # 确保类型无错
npm run dev         # 启动项目手动测试功能
```

如果有任何错误，把错误信息原样发给 AI，说明操作步骤，它能快速修正。

## 🧰 必备工具

- **快照脚本**：项目自带的 `collect_react_info.py`，每次对话前生成最新快照。
- **TypeScript 检查**：`npm run typecheck`，每次修改后必须运行。
- **浏览器控制台**：查看运行时错误，将日志完整提供给 AI。

---

## 📋 避免踩坑

1. **不要一次要求太多**：AI 输出的代码量有限，一次修改 1-2 个文件最为稳妥。
2. **始终要求完整文件**：避免手动拼凑代码，粘贴文件直接覆盖最安全。
3. **严格按顺序推进**：架构重构 → bug 修复 → 类型化 → 功能 → 注释 → 文档，不要跳跃。
4. **善用“继续下一步”**：AI 能记住当前进度，只需说“下一步”即可。
